Claude Code 활용 방식: 계획과 실행의 분리

AI 코딩 도구를 쓰면서 가장 흔한 패턴은 “프롬프트 입력 → 코드 생성 → 수정 반복”이다. 하지만 9개월간 Claude Code를 주 개발 도구로 사용한 개발자가 제시하는 방식은 다르다. 코드 작성 전에 반드시 명시적인 계획 검토 단계를 거치는 워크플로우로, Research → Plan → Annotation → Todo List → Implementation → Feedback의 순환 구조를 따른다.

핵심 원칙은 단순하다. “계획 승인 전에는 Claude에게 코드를 쓰게 하지 않는다.” AI의 실행력과 인간의 판단력을 분리하고 결합하는 접근이다.

Phase 1: Research — 깊이 있는 코드베이스 분석

모든 작업은 코드베이스 분석으로 시작한다. Claude에게 특정 폴더나 기능을 “깊이 읽고 세부적으로” 분석하도록 지시하고, 결과는 반드시 research.md 파일로 기록하게 한다.

이 문서는 Claude의 이해도를 검증하는 리뷰 표면(review surface) 역할을 한다. 오해가 있으면 이 단계에서 수정한다. 잘못된 리서치는 잘못된 계획과 구현으로 이어지므로, 가장 비용이 큰 실패 요인을 사전에 차단하는 셈이다.

예시로 캐싱 레이어 무시, ORM 규칙 미반영, 중복 로직 생성 같은 문제를 이 단계에서 예방할 수 있다.

Phase 2: Planning — plan.md 작성

리서치 검토 후, Claude에게 상세 구현 계획(plan.md)을 작성하게 한다. 실제 코드 스니펫, 수정 파일 경로, 트레이드오프 등을 포함한다. 내장된 plan 모드 대신 직접 관리 가능한 markdown 파일을 사용하는 것이 포인트다.

오픈소스의 참조 구현(reference implementation)을 함께 제공하면 Claude의 계획 품질이 크게 향상된다고 한다.

Annotation Cycle — 핵심 차별점

이 워크플로우의 가장 독특한 부분은 Annotation Cycle이다. 작성자가 plan.md를 열어 직접 인라인 주석을 추가한다.

  • 잘못된 가정 수정: “이건 PATCH여야 함”
  • 접근법 거부: “캐싱 불필요”
  • 도메인 지식 추가: “visibility 필드는 리스트 단위로 이동”

이후 Claude에게 “주석을 반영해 문서를 갱신하되, 아직 구현하지 말라”고 지시한다. 이 주석-갱신 사이클을 1~6회 반복하면서 “don’t implement yet” 명령으로 조기 실행을 방지한다.

markdown 문서가 공유 가능한 상태(state)로 작동하면서, 반복을 통해 일반적 계획이 실제 시스템에 완벽히 맞는 사양으로 진화한다.

구현 직전에는 Claude에게 세부 작업 목록(todo list)을 추가하게 해서 진행 상황을 추적한다.

Phase 3: Implementation — 기계적 실행

모든 결정이 확정된 후 표준화된 프롬프트로 실행을 지시한다.

  • “모든 태스크를 완료할 때까지 멈추지 말 것”
  • “불필요한 주석 금지”
  • “any/unknown 타입 금지”
  • “지속적 타입체크 수행”

이 단계는 계획을 그대로 실행하는 기계적 단계다. 창의적 판단은 이미 완료된 상태이므로, 실행 중 작성자는 감독자 역할로 전환해 짧고 명확한 피드백만 전달한다. 잘못된 방향으로 진행되면 git revert 후 범위를 축소해 재시도하는 것이 점진적 수정보다 효과적이라고 한다.

커뮤니티 반응과 변형 접근법

이 글은 GeekNews와 Hacker News에서 활발한 논의를 이끌었다.

서브 에이전트 구조: 하나는 계획, 하나는 구현, 또 다른 하나는 리뷰를 맡기는 방식. 블루팀/레드팀처럼 역할을 분리하면 더 명확해진다는 의견이 있다.

검증 루프 중심 접근: plan.md 대신 실행 가능한 스캐폴드와 검증 루프를 중심으로 작업하는 방식도 제시되었다. 초기에 실행 가능한 뼈대를 유지하고, 한 번에 얇은 수직 슬라이스만 추가하며, 테스트와 타입 등 검증 가능한 산출물을 강제하는 접근이다.

배치 분할: 계획이 완벽하지 않으면 처음부터 다시 해야 하는 단점을 보완하기 위해, 디자인 → 계획 → 실행을 여러 배치로 나누는 방법도 있다. 1,500줄 이하의 계획 단위로 나누면 복잡도를 관리하기 쉽다.

한편 Amazon의 Kiro, Google의 Antigravity, GitHub의 Spec Kit 같은 도구들이 이러한 계획-실행 분리를 제품화하고 있다는 점도 주목할 만하다.

핵심 정리

“깊이 읽고, 계획을 쓰고, 주석으로 다듬은 뒤, 한 번에 실행하라.”

복잡한 프롬프트나 시스템 지시 없이도, 사고(Thinking)와 타이핑(Typing)을 분리한 규율적 파이프라인으로 높은 품질을 확보할 수 있다. Research는 무지한 변경을 막고, Plan은 잘못된 변경을 막으며, Annotation은 인간의 판단을 주입한다. 최종 실행은 모든 결정이 확정된 후 자동화된 절차로 수행된다.
